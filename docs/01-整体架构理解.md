# 1. 整体架构理解

## 为什么不能直接把前后端代码丢到服务器上就完事？

本地开发时，前端用 `npm run dev` 启动开发服务器，后端用 IDE 直接运行，两者通过 Vite 的 proxy 配置互通。但部署到服务器后，没有开发服务器，需要一个"中间人"来同时处理前端页面和后端接口请求——这就是 Nginx 的角色。

## 架构图

```
用户浏览器
    │
    ▼
Nginx（监听 80/443 端口）
    │
    ├── 请求路径是 /app-api/xxx
    │       → 反向代理，转发给 Spring Boot（本地 48080 端口）
    │
    └── 其他所有路径
            → 返回 Vue 打包后的静态文件（HTML/JS/CSS）
```

## 三个核心组件的职责

| 组件 | 职责 | 运行端口 |
|------|------|----------|
| **Nginx** | 托管前端静态文件；反向代理后端 API；处理 HTTPS | 80 / 443 |
| **Spring Boot** | 处理业务逻辑、数据库操作、返回 JSON 数据 | 48080（仅本地监听） |
| **MySQL** | 存储业务数据 | 3306（仅本地监听） |

## 关键概念

### 反向代理

Nginx 接收到 `/app-api/` 开头的请求后，不会自己处理，而是转发给后端的 Spring Boot，再把后端的响应原样返回给浏览器。对浏览器来说，它始终只和 Nginx 通信，不知道后端的存在。

**好处**：
- 后端的 48080 端口不需要对外暴露，减少攻击面
- 可以在 Nginx 层统一处理 HTTPS、缓存、压缩等

### 为什么后端用 48080 而不是 80？

- 80 端口已经被 Nginx 占用
- 48080 是一个高位端口，不对外开放，只在服务器内部由 Nginx 转发访问
- 这样即使有人扫描服务器端口，也无法直接访问后端接口

### Vue 的 history 模式

Vue Router 使用 history 模式时，URL 看起来像普通路径（如 `/products/aiot`），但实际上这些路径在服务器上并不存在对应的文件。如果用户直接访问或刷新这类 URL，Nginx 会返回 404。

解决方法是配置 `try_files`：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

含义：先找真实文件 → 找不到就返回 index.html → 由 Vue Router 在前端处理路由。
